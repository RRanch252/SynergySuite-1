01 CONTEXT

Every new dashboard panel used to be a small project.

The legacy approach treated each panel as bespoke work—custom queries written once and forgotten, hard-coded layouts, no schema awareness, no review gate, and no way to roll a change from one environment to the next without manual intervention.

Before

One-off effort

Every new panel meant net-new SQL, net-new layout code, net-new frontend integration—engineering time per customer request.

Before

No review gate

Queries shipped without any structured review for performance, permission scope, or convention adherence. Issues found in production.

Before

Brittle promotion

Moving a panel from staging to production was a manual config copy with database-ID hand-mapping. Error-prone, slow, undocumented.

02 SYSTEM DESIGN

Four services. One contract.

The new system separates concerns cleanly: a query engine, a layout engine, the customer-facing application, and the internal management surface. Every service speaks the same configuration language.

Four services, cleanly separated by responsibility. The Analytics Service owns data; the Format Service owns layout. The customer app and the internal portal both consume the same APIs—nothing the internal team configures is unreachable for end users, and nothing customers see was built outside the documented contract.

Why this separation matters

Data logic and layout logic evolve at different speeds. Queries change when business definitions change; layouts change when users want a different visual. By keeping them in separate services with separate deployment cycles, neither blocks the other.

Waterfall inheritance

Dashboards cascade: person overrides company, company overrides corporate, corporate overrides default. One configuration change at the corporate level reaches every user beneath it without touching individual records.

03 THE REPEATABLE PATTERN

Six steps. Every panel. Every time.

The point of the new platform isn't a single dashboard—it's a workflow that turns "I want a panel that shows X" into a deployed, validated, permission-aware feature with predictable engineering cost.

STEP 01

Author Query

Monaco editor in the Service Portal. Schema-aware autocomplete on every table and column.

STEP 02

AI Review

The QUERY_REVIEW agent checks security, performance, schema correctness, conventions.

STEP 03

Configure Panel

Pick a panel type from the library. Structured form, with raw-JSON toggle for power users.

STEP 04

Export Bundle

Single JSON file with logical references—no fragile database IDs to break across environments.

STEP 05

Promote

Import the bundle to staging, then production. Same JSON, same result, every time.

STEP 06

Apply via Inheritance

Attach at the level you want—one customer, one corporate parent, or the global default.

The same workflow ran for every one of the twelve panel types we shipped. It will run unchanged for the next twelve. The cost of each new panel is now the cost of writing one SQL query plus selecting a panel type—not the cost of building a feature.

06 THE AI FOUNDATION

The dashboard work is also AI infrastructure work.

Every component we needed for safe, useful AI inside SynergySuite is something we needed anyway for the dashboard overhaul. The two initiatives share a single substrate.

Two strategic outcomes. One foundation. The dashboard overhaul is not parallel to the AI initiative—it is the prerequisite for it being safe and useful.

Mapped, capability by capability

Every piece of infrastructure listed below was built for the dashboard. Each of them is also exactly what a production-grade conversational AI inside SynergySuite would need. We've been investing in both at once.

  • Schema introspection
    Powers Monaco autocomplete in the SQL editor
    Lets the LLM know what tables and columns actually exist
  • QUERY_REVIEW agent
    Validates queries before they're registered
    Validates AI-generated SQL before it ever runs against a customer DB
  • Query registry
    Permission-aware catalogue of approved queries
    A pre-vetted safe-set the chatbot can compose from, no ad-hoc SQL needed
  • Virtual foreign keys
    Surfaced in schema endpoint for editor hints
    Teach the model how entities relate without DDL constraints
  • Permission enforcement
    Customer A never sees Customer B's data on a dashboard
    Customer A's chatbot literally cannot answer about Customer B's data
  • 12-component render library
    The visual vocabulary of every dashboard
    The chatbot picks one to display each answer—a chart, a table, a single number
07 DIRECT TO INVESTOR INTEREST

Conversational AI, safely—because the floor is already built.

"The hardest part of putting a chatbot in front of a customer database isn't the language model. It's everything underneath: knowing the schema, enforcing permissions, validating generated SQL, and rendering the result so it's actually useful. We've already built that floor."

The risk profile of a naïve chatbot

An AI chatbot wired directly to a production database is a serious liability. It can hallucinate columns, write expensive queries that take down shared infrastructure, leak data across permission boundaries, and present results without any verification step. Most attempts at "AI for business data" fail on one of these.

Putting a chatbot in front of SynergySuite without addressing all four would be reckless—and the user experience would be inconsistent enough that customers would lose trust quickly.

How our foundation neutralises each risk

Hallucinated columns → the schema endpoint gives the model ground truth. Autocomplete already proves the loop works.

Expensive queries → the QUERY_REVIEW agent enforces performance conventions on every generated query, before execution.

Permission leakage → the query registry is permission-aware. The chatbot can only compose from a customer's approved set.

Useless results → the 12 panel types are the rendering vocabulary. Every answer can be a real visualisation, not a wall of text.

The strategic claim

The dashboard overhaul isn't a detour. It's the runway.

Every quarter we ship on the dashboard pattern is a quarter that compounds on the chatbot's safety, accuracy, and usefulness. We can ship a credible conversational AI sooner—and one that we can stand behind—precisely because we invested in this infrastructure first. The alternative path (build the chatbot first, retrofit safety later) is how teams ship something embarrassing.

08 WHAT'S NEXT

The path from here to a conversational, agentic SynergySuite.

Near-term work continues to grow the library and deepen the AI tooling. Each step is independently shippable, each step compounds with the rest, and each step moves the chatbot vision closer without requiring a separate initiative.

In progress · Segment Comparison Panel
A 13th panel type: side-by-side segment delta analysis

Variable segment count, configurable comparison reference, inline delta rendering, currency-aware formatting from company context. Design is done; implementation is phased across all four codebases with cross-project verification.

Next · Phase K
Controlled ad-hoc query runner

SELECT-only enforcement, mandatory audit logging, optional AI pre-execution review, result-size limits. Replaces direct database client access for internal users—and is the natural execution surface for chatbot-generated queries later.

Near future · Conversational layer
Natural-language → registered query

The chatbot does not write SQL from scratch. It picks from the permission-aware query registry, parameterises it, and explains the choice. The output is rendered through the panel library. Safety is structural, not aspirational.

Future · Proactive insights
Agents that watch, not just answer

Once we have the agent infrastructure and the rendering library, the next move is agents that surface anomalies before being asked. "Outlet 47's labour cost trended up 18% this week, here's the comparison panel." The dashboard, but anticipatory.

Future · Conversational dashboard authoring
"Build me a dashboard that…" — by the customer themselves

The Service Portal workflow, expressed as a conversation. Customer admins describe the dashboard they want; the agent proposes a layout from the existing library, validated by the same review agent. The internal team becomes a quality gate, not a request queue.
